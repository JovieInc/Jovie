# Artist Profiles Landing Page

## Narrative Summary

This page sells Jovie Artist Profiles as the single music-native link that adapts to release moments, show moments, and fan-capture moments without drifting into a generic creator website pitch. The narrative should move from category definition to product proof to objection handling, then close with a simple claim action.

## Locked Section Order

1. Hero
2. Adaptive Thesis + Mode Switcher
3. Outcomes Carousel
4. Capture Every Fan
5. Opinionated by Design
6. Feature / Spec Wall
7. How It Works
8. Social Proof
9. FAQ
10. Final CTA

## Section Spec

### 1. Hero

- Purpose: Set the category frame and convert immediately.
- User question: Why is this better than a generic bio link?
- Why it exists: The page needs to establish that Jovie is purpose-built for artists, not a general link page with music pasted into it.
- Visual treatment: Premium hero with the phone as the core visual device, the `jov.ie/you` signature detail, restrained atmosphere, and a quiet proof whisper if real proof is available.
- Key interaction: Primary CTA only.
- Copy goal: Keep the current hero concept nearly verbatim.
- Proof / CTA presence: CTA is required. Proof whisper is optional and must stay quiet.

### 2. Adaptive Thesis + Mode Switcher

- Purpose: Prove the core thesis that one profile can serve many moments.
- User question: What does "adapts" actually mean?
- Why it exists: This is the strategic center of the page and the cleanest answer to Linktree, Linkfire, Feature.fm, and ToneDen style objections.
- Visual treatment: Pinned phone with a mode rail and concise surrounding copy.
- Key interaction: Switch between Upcoming Release, Release Day, Touring, and Live Support.
- Copy goal: Make countdowns, smart routing, local shows, and live support feel inevitable rather than feature-listed.
- Proof / CTA presence: No hard CTA. Soft proof of deep-linkable modes only.

### 3. Outcomes Carousel

- Purpose: Show fan outcomes instead of product modules.
- User question: What changes for the fan?
- Why it exists: After the thesis lands, the page needs to translate product behavior into immediate outcomes.
- Visual treatment: Horizontal product cards anchored by phone crops or surface snippets.
- Key interaction: Swipe or scroll through cards.
- Copy goal: Frame listen, show, support, subscribe, and continuity as outcomes.
- Proof / CTA presence: No CTA.

### 4. Capture Every Fan

- Purpose: Sell owned audience and relationship capture.
- User question: Do I just get clicks, or do I get people?
- Why it exists: This is one of the sharpest differentiators from static bio links and generic smart link tools.
- Visual treatment: Notification or capture state plus lightweight, real product proof such as subscription state or profile-follow flow.
- Key interaction: None required.
- Copy goal: Explain opt-in once, release alerts, and identity capture without sounding like a CRM pitch.
- Proof / CTA presence: Soft proof only.

### 5. Opinionated by Design

- Purpose: Answer the customization objection directly.
- User question: Can I tweak everything?
- Why it exists: Artists coming from Linktree-style tools will expect templates and theme builders. Jovie should reject that tradeoff on purpose.
- Visual treatment: Calm product render paired with short principles, not a big comparison layout.
- Key interaction: None.
- Copy goal: No theme builder, no template maze, conversion-first defaults.
- Proof / CTA presence: No CTA.

### 6. Feature / Spec Wall

- Purpose: Satisfy technical artists and DJs who want the detailed product truth.
- User question: What exactly does this do?
- Why it exists: The page needs a compact proof surface for speed, routing, modes, releases, shows, support, capture, notifications, and QR without breaking pacing.
- Visual treatment: Dense compact grid, not a flashy bento and not a comparison table.
- Key interaction: Hover or reveal is optional, but the base state must communicate clearly without interaction.
- Copy goal: Short, concrete proof tiles for speed, routing, modes, releases, shows, pay, capture, notifications, QR, and restrained analytics.
- Proof / CTA presence: Optional inline text link to claim only.

### 7. How It Works

- Purpose: Reduce setup anxiety.
- User question: How fast can I get live?
- Why it exists: Zero setup is one of the strongest objections to close before the user reaches the end of the page.
- Visual treatment: Compact three-step strip with one line per step.
- Key interaction: None.
- Copy goal: Claim -> Connect or import -> Share.
- Proof / CTA presence: Optional CTA only on the final step if needed.

### 8. Social Proof

- Purpose: Confirm that the product is real and already grounded in artist workflows.
- User question: Who is already using or validating this?
- Why it exists: Social proof should support trust, not lead the story.
- Visual treatment: Real profile cards first, logos second, founder quote third.
- Key interaction: Optional carousel only if real artist quotes exist.
- Copy goal: Real product proof, real artist workflows, real music context.
- Proof / CTA presence: Proof only.

### 9. FAQ

- Purpose: Handle search and objection cleanup without bloating the page.
- User question: What objections still remain?
- Why it exists: This page needs to close common comparison, setup, and use-case questions while staying tight.
- Visual treatment: Compact accordion with four items maximum.
- Key interaction: Expand and collapse.
- Copy goal: Direct answers on Linktree difference, setup speed, release-day use, and shows / support / capture behavior.
- Proof / CTA presence: No CTA.

### 10. Final CTA

- Purpose: Close on the thesis and convert.
- User question: Am I ready to claim this?
- Why it exists: The end of the page should not introduce new information. It should restate the thesis and ask for the claim.
- Visual treatment: Clean repeat phone or URL treatment with the `jov.ie/you` signature.
- Key interaction: Primary CTA.
- Copy goal: One link for every release, show, and fan action.
- Proof / CTA presence: CTA is required.

## Cuts, Merges, And Avoids

- Cut the standalone "One profile." interstitial.
- Merge the thesis and mode switcher into one section.
- Do not keep a separate notifications-only chapter.
- Do not use a generic comparison table.
- Do not lead with a founder-first social proof block.
- Do not create a standalone analytics section.
- Do not position Jovie as a general creator website tool.
- Do not recommend theme builders, templates, or open-ended customization.
- Do not use fake stats, invented testimonials, or generic SaaS claims.

## Final Recommendations

- FAQ belongs, but only as four tight questions near the bottom.
- How It Works belongs, but only as a compressed low-page three-step section.
- The Feature / Spec Wall should sit after Opinionated by Design and before How It Works.

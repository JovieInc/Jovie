# Linear ↔ AgentOS Roadmap Sync Model

**Status:** Canonical contract (JOV-1930)
**Owner:** AgentOS / Roadmap System project
**Consumers:** `/roadmap` parser (JOV-1932), agent-brief generator (JOV-1933), AgentOS Linear app user (JOV-1934), all downstream agent runtimes (Hermes, Ruflo, Vercel Workflow)
**Related schema:** `apps/web/lib/agent-os/artifact.ts` (`AgentRunArtifact`)

This document is the authoritative contract for how state flows between Linear (the durable roadmap) and the `agentos/` repo mirror (execution detail). Every future AgentOS issue depends on this split. If a field's location is ambiguous, this file resolves it.

---

## 1. The split

| Layer | Lives in Linear | Lives in `agentos/` repo |
|---|---|---|
| What to build | Initiative, Project, Issue title, parent/child, blocking relations | — |
| Why / scope | Issue description (≤ 2 paragraphs, scope only) | `agentos/roadmap/<project>.md` (full architecture, data models, edge cases) |
| Priority | Linear `priority` (0–4) | — |
| Status | Linear `state` (Triage / Backlog / Todo / In Progress / In Review / Done / Canceled / Duplicate) | — |
| Owner | Linear `assignee` + `delegate` (for AgentOS app user) | — |
| Approval gates | Linear comments + `human-review-required` label | `humanGate` slot in agent brief (mirrors Linear) |
| Decisions / discussion | Linear comments | `agentos/memory/decisions.md` (durable, append-only mirror of approved decisions) |
| Taste rules | — | `.claude/rules/*.md`, `agentos/memory/design-taste.md`, `agentos/memory/rejected-directions.md` |
| Agent instructions | — | `agentos/agents/<agent>.md`, `agentos/roadmap/<project>.md` |
| Repo file references | Issue description may link a path | `repo_specs[]` in agent brief |
| Generated artifacts | Linear comment summary + attachment URL | `agentos/runs/<kind>/<run-id>/` (full payload) |
| Cached mirror of Linear | — | `agentos/roadmap/backlog.json` |

**Rule of thumb:** if a human reorders the roadmap, the change happens in Linear. If an agent needs to know *how* to execute, the change happens in the repo.

## 2. Sync direction + write authority

Two independent flows, two distinct write authorities, asymmetric on purpose.

### 2.1 Linear → repo (read-only mirror)

- A scheduled job (`/roadmap sync`, see JOV-1932) reads Linear via MCP/GraphQL and writes `agentos/roadmap/backlog.json`.
- The mirror is **regenerated**, never hand-edited.
- `agentos/roadmap/backlog.json` is committed so agents in fresh worktrees have a deterministic starting point without a network round-trip.
- Cadence: cron every 15 min in production HUD; on-demand via `/roadmap sync` from any session.
- Failure mode: stale mirror is acceptable for ≤ 1 hour. Drift detection (§5) catches longer drift.

### 2.2 Repo → Linear (scoped writes)

Writes back to Linear are split by identity:

| Identity | Allowed writes | Forbidden |
|---|---|---|
| Human (Tim, reviewers) | All Linear operations | — |
| **AgentOS Linear app user** (JOV-1934) | Comments on agent-owned issues; `add/remove` labels on agent-owned issues; status transitions on agent-owned issues (Todo → In Progress → In Review → Done only); attaching links/PRs | Editing title; editing description; changing priority; changing assignee; transitioning to Canceled/Duplicate; any write on human-owned issues |
| Other agents (Hermes, Ruflo, CI) | Must funnel through the AgentOS app user — no direct Linear API tokens issued to runtimes | — |

### 2.3 "Agent-owned" — concrete definition

An issue is **agent-owned** when **all three** are true:

1. It carries the label `agentos` (every issue created under the AgentOS initiative is auto-labeled).
2. Its `delegate` is the `AgentOS` Linear app user (set by `/roadmap add` or `/roadmap expand`).
3. It does **not** carry the label `human-review-required`.

If any of those three is false, the AgentOS app user is read-only on the issue. The label `human-review-required` is the canonical kill switch — applying it instantly demotes the issue to human-only writes, even mid-run.

## 3. `backlog.json` schema

`agentos/roadmap/backlog.json` is the materialized view of Linear state used by every agent run. It is regenerated by `/roadmap sync` and committed.

### 3.1 TypeScript shape

```ts
// agentos/roadmap/backlog.types.ts (to be created in JOV-1932)

export interface RoadmapBacklog {
  /** ISO datetime of the last successful Linear → repo sync */
  syncedAt: string;
  /** SHA of the Linear MCP/GraphQL revision used to build this snapshot, if available */
  sourceRevision: string | null;
  /** AgentOS initiative-level metadata */
  initiative: {
    id: string;            // Linear initiative ID
    name: string;          // e.g. "AgentOS"
    url: string;
  };
  projects: RoadmapProject[];
  issues: RoadmapIssue[];
}

export interface RoadmapProject {
  id: string;              // Linear project ID
  name: string;            // e.g. "Roadmap System"
  slug: string;            // kebab-case used for repo file matching
  status: 'planned' | 'started' | 'paused' | 'completed' | 'canceled';
  url: string;
  /** Path (relative to repo root) of the spec mirror, if one exists */
  specPath: string | null; // e.g. "agentos/roadmap/roadmap-system.md"
}

export interface RoadmapIssue {
  id: string;              // Linear short ID, e.g. "JOV-1930"
  uuid: string;             // Linear UUID (for MCP write calls)
  title: string;
  url: string;

  state: {
    name: string;          // "In Progress"
    type: 'triage' | 'backlog' | 'unstarted' | 'started' | 'completed' | 'canceled';
  };
  priority: 0 | 1 | 2 | 3 | 4;   // 0=None, 1=Urgent, 2=High, 3=Medium, 4=Low
  assignee: { id: string; name: string } | null;
  delegate: { id: string; name: string } | null;
  labels: string[];

  projectId: string | null;
  parentId: string | null;       // Linear parent issue (sub-issue link)
  blockedBy: string[];           // Linear short IDs
  blocks: string[];

  /** Computed by the sync job from §2.3 */
  agentOwned: boolean;
  humanReviewRequired: boolean;

  /** Repo paths referenced in the issue description (links and inline mentions) */
  repoFileRefs: string[];

  /** Derived: matches an attached PR via `<!-- linear-issue-id:JOV-XXXX -->` */
  pullRequestUrl: string | null;

  createdAt: string;
  updatedAt: string;
  /** ISO datetime this row was last refreshed from Linear */
  lastSyncedAt: string;
}
```

## 4. Agent brief shape

The brief is the per-run input handed to a runtime (Hermes/Ruflo/Vercel Workflow) when an agent picks up a Linear issue. It is generated by JOV-1933 and consumed by JOV-1926-class gate enforcement. The brief's run-result (status, gates, costs) is reported back via `AgentRunArtifact` (see `apps/web/lib/agent-os/artifact.ts`); the brief is the *input*, the artifact is the *output*. Their `linearIssueId` and allowed/forbidden action vocabularies must align.

### 4.1 TypeScript shape

```ts
// agentos/roadmap/brief.types.ts (to be created in JOV-1933)

import type {
  AgentRunAction,
  AgentRunKind,
  AgentRunModelRoute,
} from '@/lib/agent-os/artifact';

export interface AgentBrief {
  /** Stable ID for this brief; the resulting AgentRunArtifact.id mirrors this */
  id: string;                          // e.g. "brief-jov-1930-2026-05-08T17:30:00Z"
  schemaVersion: 1;

  /** Linear context */
  linearProject: { id: string; name: string; slug: string };
  currentIssue: {
    id: string;                        // "JOV-1930"
    title: string;
    url: string;
    description: string;               // raw Markdown from Linear
    priority: 0 | 1 | 2 | 3 | 4;
    state: string;
    labels: string[];
  };

  /** Other Linear issues this brief depends on, for context-loading */
  dependencies: {
    blockedBy: Array<{ id: string; title: string; state: string }>;
    blocks: Array<{ id: string; title: string; state: string }>;
    siblings: Array<{ id: string; title: string; state: string }>; // same project
  };

  /** Repo paths the agent MUST read before editing */
  repoSpecs: string[];                 // e.g. ["agentos/roadmap/SYNC_MODEL.md", "apps/web/lib/agent-os/artifact.ts"]

  /** Scoped rule files the agent MUST honor */
  tasteRules: string[];                // e.g. [".claude/rules/ui.md", ".claude/rules/security.md"]

  /** gstack skills recommended for this kind of work */
  gstackSkills: string[];              // e.g. ["/qa --exhaustive", "/review", "/ship"]

  /** Capability vocabulary — references AgentRunArtifact action enum */
  kind: AgentRunKind;
  modelRoute: AgentRunModelRoute;
  allowedActions: AgentRunAction[];
  forbiddenActions: AgentRunAction[];
  humanApprovalRequired: boolean;
  humanApprovalReason: string | null;

  /** What success looks like; lifted into the AgentRunArtifact summary */
  successCriteria: string[];           // bullet list, agent-readable
  verificationGates: string[];         // names from AGENT_RUN_GATE_EVIDENCE_NAMES

  /** Provenance */
  generatedAt: string;
  generatedBy: 'agentos-app-user' | 'human';
  backlogSyncedAt: string;             // pin to the backlog.json snapshot used
}
```

### 4.2 Action vocabularies, not free text

`allowedActions` and `forbiddenActions` MUST draw from `AGENT_RUN_ACTIONS` in `apps/web/lib/agent-os/artifact.ts`. The brief generator must reject any custom action string. This keeps brief-time guardrails and run-time artifact gates symmetric.

### 4.3 Required `forbiddenActions` minimums

Every brief must include at minimum: `mutate_production_data`, `change_auth`, `change_billing`, `change_security` in `forbiddenActions` unless `humanApprovalRequired` is `true` AND a human gate is approved on the linked Linear issue. The brief generator enforces this.

## 5. Drift detection + resolution

Drift = Linear and `backlog.json` disagree on a field that should match.

### 5.1 Detection

- `/roadmap sync --check` (read-only) runs the sync logic into memory and diffs against the on-disk `backlog.json`. Exits non-zero if a tracked field drifts.
- A daily cron (HUD job, owner: AgentOS app user) runs `--check`. Drift produces a Linear comment on the affected issue and a HUD alert. Drift on the `backlog.json` itself produces an `agentos` initiative-level comment.
- Tracked drift fields: `state`, `priority`, `assignee`, `delegate`, `labels`, `projectId`, `parentId`, `blockedBy`, `blocks`. Untracked: `description`, `title` (humans edit those freely; agents must re-read from Linear at brief time, never trust the mirror).

### 5.2 Resolution policy

**Linear wins, always.** Repo state is a derived view; if it drifts, regenerate the mirror. There is no field where the repo wins over Linear — that would re-introduce the drift class this whole architecture exists to prevent.

Two narrow exceptions:

1. `backlog.json.syncedAt` and `lastSyncedAt` are repo-only metadata; Linear has no opinion.
2. Repo path references in `repoFileRefs` are *derived* from Linear description text. If the description mentions `apps/web/lib/foo.ts` but that file doesn't exist in the repo, the sync job warns but does not "fix" Linear — humans either update the description or create the file.

### 5.3 Repair

`pnpm /roadmap sync --force` regenerates `backlog.json` from Linear unconditionally. Commit the result on its own PR (`chore(agentos): resync backlog`). This must be runnable by the AgentOS app user without human review (low-risk, deterministic, auto-mergeable).

## 6. Open questions

These need a human decision before JOV-1932/1933 implementation lands. Listed; not answered.

1. **Sync cadence**: 15 min cron (proposed) vs. on-demand only vs. webhook-driven. Linear webhooks would be lower-latency but require a public endpoint — see `.claude/rules/security.md` "Public/Webhook Coordination Must Be Durable."
2. **`backlog.json` location**: committed in repo (proposed, deterministic) vs. generated to `.next/cache`/Redis (smaller diff noise, but breaks fresh-worktree determinism). Lean: keep committed; suppress diff noise via `.gitattributes` `merge=union` if it becomes painful.
3. **Per-project spec file naming**: `<project-slug>.md` vs. `<project-id>.md`. Slug is human-readable; ID is rename-safe. Lean: slug, with frontmatter pinning the Linear project ID for resolution stability.
4. **Multiple Linear teams**: contract today assumes single team `Jovie` (id `bdc09edc-...`). If AgentOS expands across teams, the schema needs a `team` field on every issue and project. Punt until needed.
5. **Brief versioning**: when `AgentBrief` shape changes, do we re-generate all in-flight briefs or pin runtime to v1? Lean: pin runtime to `schemaVersion`, refuse to execute mismatched briefs.
6. **AgentOS app user's status-transition vocabulary**: `Todo → In Progress → In Review → Done` is proposed. Should the app user also be allowed to move issues to `Backlog` (e.g. when a brief is rejected pre-execution)? JOV-1934 should resolve this when provisioning.
7. **Auto-cancel stale agent-owned issues**: should drift detection cancel issues whose linked PR has been closed without merge for > 14 days, or only flag them? Lean: flag only; cancellation is a human-owned transition.

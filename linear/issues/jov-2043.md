# JOV-2043: gbrain: autopilot daemon holds PGLite lock during sync — add lock arbitration

- **Updated:** 2026-05-31T05:03:36.208Z

## Symptom

Running `gbrain sync --skip-failed` returns:

```
GBrain: Timed out waiting for PGLite lock.
```

Reproduces consistently when the gbrain autopilot daemon is running and any other gbrain CLI tries to write.

## Root cause

PGLite is single-writer. The autopilot daemon holds an exclusive lock for its dream cycles. CLI invocations (`gbrain sync`, `gbrain put`, etc.) wait until the lock is released or time out.

Existing memory at `feedback_gbrain_autopilot_lock.md` captured this: "autopilot daemon holds PGLite lock — use cron dream cycles when MCP is also a client."

## Fix candidates

1. Document: warn users to stop autopilot before manual `gbrain sync`. Workaround, not a fix.
2. Detect: gbrain CLI checks for autopilot heartbeat at startup; refuses with a clear message ("autopilot is running and holds the DB lock; stop it first or wait for the next dream window").
3. Architectural: PGLite lock is a known limitation; consider Supabase migration for multi-writer scenarios.

## Suggested first move

Option 2 — detection + clear error. Cheap, doesn't require migration.

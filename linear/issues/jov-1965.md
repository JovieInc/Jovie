## Follow-up

Adopt the action-queue spine from the ChatGPT "Ops Cockpit" proposal that was deferred during the 2026-05-08 /autoplan review. Today the HUD is a tile dashboard; the v2 turns each signal into an actionable queue item.

Concrete scope:

* **New** `/api/hud/actions` endpoint returning `OpsAction[]` with shape `{ id, severity, source, title, summary, status, owner, recommendedAction, requiresApproval, linkedPr, linkedDeploy, linkedIssue, createdAt, updatedAt }`. Statuses: `queued | investigating | fixing | verifying | ready-for-pr | blocked | shipped | dismissed`.
* **Postgres-backed** `ops_events` table (Drizzle migration, additive). Single source of truth — NOT a local-file event source. Event types: `deploy.started/succeeded/failed`, `pr.opened/stale`, `agent.started/blocked/completed`, `metric.threshold_crossed`, `approval.required/granted/denied`.
* **Replace tile display with Attention Queue** — sorted by severity × business impact. Use existing `EntitySidebarShell` for the detail drawer (what happened / why it matters / owner / suggested next action / linked PR/issue/runbook/deployment).
* **Approval-gate chips** — explicit UI for: production deploys, billing changes, auth changes, DB migrations, outbound messaging, spend. Map to the existing entitlement system (`@/lib/entitlements/server`).
* **Hermes summary panel** — top 3 risks, top 3 next actions, blocked work, stale work, human approvals needed, estimated business impact. Reuses the existing dispatch endpoint (`/api/hud/ai-ops/dispatch`).
* **Kiosk wallboard reuses the same data** — `/hud-tv?kiosk=TOKEN` consumes the same `/api/hud/actions` endpoint and renders the queue at TV scale (no dispatch UI, read-only).

## Why it matters

The current /hud (and the new /app/hud landing in PR 2) is a status wallboard. It tells you what is, not what to do about it. As Jovie operations get more complex (multi-environment deploys, more agent PRs, more cron jobs, real revenue), the team needs an operational decision queue, not another dashboard. Both /autoplan voices independently flagged adding more dashboards as the central architectural risk; this is the productive direction that addresses the same concern.

API-everything: every signal flows through `/api/hud/actions` and is queryable, transitionable, and auditable. Don't-build-twice: kiosk wallboard and shell cockpit consume the same endpoint.

## Triage question

Does Jovie's current operational tempo (deploy frequency, agent PR volume, incident rate) justify an action-queue model, or are we still at the "tile dashboard is enough" phase? If unsure: ship the v2 in two cuts — (1) `OpsAction` API + Attention Queue list (\~3 days), (2) approval gates + Hermes summary panel (\~3 more days). Each cut must be justified by usage telemetry from PR 2's `hud.shell_viewed` / `hud.dispatch_invoked` events showing real engagement.

## Dependency

## Acceptance criteria (if implemented)

* `OpsAction` Drizzle schema + migration deployed to dev and main without `db.transaction()`.
* `GET /api/hud/actions` returns `OpsAction[]`; `POST /api/hud/actions/:id/transition` enforces approval gates via existing entitlement system.
* Attention Queue list replaces (or augments) the current tile grid on `/app/hud` shell mode.
* Detail drawer uses existing `EntitySidebarShell`, no parallel drawer primitive.
* Approval-gate chips render for any action where `requiresApproval=true`.
* Kiosk wallboard at `/hud-tv` consumes the same endpoint.
* Telemetry: `hud.action_clicked`, `hud.approval_granted`, `hud.action_dismissed` events.
* E2E test: an action lifecycle from `queued` → `shipped` with at least one approval gate transition.

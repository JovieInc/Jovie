## Follow-up

[JOV-2127](https://linear.app/jovie/issue/JOV-2127/ops-approval-queue-add-explicit-action-buttons-per-item-open-pr-open) shipped Open PR + Open Linear action buttons on `WorkflowRunRow`. The approve/reject buttons were deferred because no server action exists to mutate `humanGate.status` on an artifact. The `AgentRunArtifact` schema has the `humanGate` shape, but there is no API route or server action that accepts an approve/reject decision and persists it (e.g., through the WDK runtime or a DB write).

What is needed:

1. A server action or API route `POST /api/admin/agent-os/workflows/runs/[runId]/gate` (or similar) that accepts `{ decision: 'approved' | 'rejected', reason?: string }`, verifies admin entitlement, and writes the decision durably.
2. The `WorkflowRunRow` `onApprove`/`onReject` props plumbed into `ApprovalQueuePanel` with a toast confirmation on success.
3. Optimistic UI update so the row transitions immediately without a full page refresh.

## Why it matters

The Approval Queue currently has no action path — admins can see items waiting for approval but cannot act on them without leaving the admin panel. This is the remaining gap from [JOV-2127](https://linear.app/jovie/issue/JOV-2127/ops-approval-queue-add-explicit-action-buttons-per-item-open-pr-open).

## Acceptance criteria or triage question

Decision: what backing store should `humanGate.status` write to? WDK runtime signal, Neon DB row, or in-memory (ephemeral)? Resolve this before implementing — the answer determines the route handler design and whether optimistic UI is safe.

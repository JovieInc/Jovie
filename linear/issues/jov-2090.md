## Follow-up

1. `apps/web/components/providers/clerkAvailability.ts:getClerkJSUrl()` re-implements the same Clerk-publishable-key → FAPI-host decoding that now lives in `apps/web/lib/auth/decode-fapi-host.ts`. The duplicate path only handles `pk_live_` whereas the canonical helper handles both `pk_live_` and `pk_test_`. Both should call the same helper.
2. `apps/web/lib/security/content-security-policy.ts` still includes `https://clerk.jov.ie` in both `scriptSrcSuffix` and `connectSrcBase`. Per `apps/web/lib/auth/README.md`, `clerk.jov.ie` is dead as a public URL — Clerk FAPI traffic now goes through the `/__clerk` path proxy on the same origin. The CSP entries are stale and confuse the security review.

Both files are outside the worcester-auth-core HOT ZONE (`components/providers/` and `lib/security/`), so they were not changed in [JOV-2061](https://linear.app/jovie/issue/JOV-2061/p0-fix-clerk-proxyruntime-auth-errors-in-production).

## Why it matters

* Duplication: a future Clerk key format change has to be applied in two places.
* Stale CSP: makes the policy harder to reason about and lengthens the header for no benefit.

## Classification

Candidate

## Acceptance criteria or triage question

Pickup agent should:

* Update `getClerkJSUrl()` to call `decodeFapiHostFromPublishableKey()`.
* Confirm with the security owner that `https://clerk.jov.ie` can be removed from `script-src` and `connect-src`, then remove it. Add a regression test that asserts the dead host is absent from the rendered CSP.

# JOV-1837: CI: structurally fix main-branch unit-test runtime (move --coverage off the gating run)

## Background

Follow-up to [https://github.com/JovieInc/Jovie/pull/8052](<https://github.com/JovieInc/Jovie/pull/8052>), which is the band-aid:

* bumped `ci-unit-tests` `timeout-minutes` from 20 → 30
* widened the shard matrix from 4 → 6

That PR clears the `Main CI health monitor` alert and gets main CI concluding `success` again, but it doesn't fix the underlying problem: **vitest with** `--coverage` on `ubuntu-latest` runners is too slow to be on the gating main-branch run. With the band-aid in place, runtime is back below the ceiling — but the suite will continue to grow and we'll be back here.

## Evidence

5 consecutive main pushes prior to PR #8052 hit `timeout-minutes: 20` flat, conclusion `cancelled`:

* 25410447063 — sha a8734d21c
* 25410966000 — sha 62d5e841f
* 25411451301 — sha 42e640bbb (one shard squeaked through at 19m26s)
* 25411970322 — sha 882efc193
* 25413205545 — sha 76fbe3451 (most recent before fix)

Logs from these runs show vitest still actively printing passing tests right up until `##[error]The operation was canceled.` at exactly T+20:00 — so this is a runtime budget issue, not a hung process.

The trigger: main pushes run with `--coverage` (line 2098 of `.github/workflows/ci.yml`); PRs don't. Coverage instrumentation adds \~25–40% wall-clock. PR CI was passing fine; only main was failing.

Despite the cancellations, **production was actually deploying correctly** — `Promote to Production`, `Post-Deploy Smoke (Production)`, `Sentry Error Gate`, and `Lighthouse CI (Production)` all `success` in run 25413205545. The cancellation was dashboard-only, but it fired the health alert and degraded the merge-queue signal.

## Proposed structural options (decide one)

1. **Move** `--coverage` to a dedicated nightly job — main CI matches PR CI speed, coverage trend is still tracked, no per-merge cost.
2. **Move** `--coverage` to a separate non-gating job that runs in parallel with the gating shards — coverage doesn't block deploys; if it's slow, only the Codecov upload is delayed.
3. **Keep** `--coverage` on main but use `vars.CI_FAST_RUNNER` (larger runner) for `ci-unit-tests` only. Costs more per run but cuts wall-clock.
4. **Profile the suite** with `--reporter=verbose` and split out the 5 slowest test files into their own shard or move them to integration/nightly.

These are not mutually exclusive (e.g., 1 + 4 is plausible). Picking the right combo depends on:

* how much we value coverage-on-every-merge vs. deploy speed
* whether we're willing to spend more on larger runners
* whether the slowest tests are slow because they're testing slow things or because they're poorly written

## Acceptance criteria

- [ ] Main-branch CI median runtime for `ci-unit-tests` back under 15 minutes per shard
- [ ] Coverage trend remains visible in Codecov (daily granularity is fine)
- [ ] Workflow timeout for `ci-unit-tests` can be reduced back to ≤20 minutes without flake risk
- [ ] PR #8052 band-aids (timeout=30, shards=6) are revisited — likely revert one or both once root cause is fixed

## Files

* `.github/workflows/ci.yml` — `ci-unit-tests` job (lines \~2012–2131); main-branch coverage flag at line 2098
* `apps/web/vitest.config.ts` (or equivalent) — coverage configuration
* `.github/workflows/nightly-tests.yml` — possible new home for coverage

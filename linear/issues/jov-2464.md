## Source

* Current issue: ad-hoc (CI/Jovie-bot autoplan, 2026-05-18)
* Source plan: R2 item 1 (resolution: not actually blocking new runs, but cancel API returns 500 — GH-side bug)

## Follow-up

9 workflow runs from 2026-05-15 are wedged in `queued` state on branches `codex/jov-2179-consent-cookie-policy` and `codex/jov-2199-audience-visit-persistence`. PRs #8706 and #8705 are MERGED. Branches are gone. Runs persist:

```
25907297234  Fork PR Gate    codex/jov-2179
25907296265  Fork PR Gate    codex/jov-2179
25907192193  Agent Pipeline  main (workflow_run from codex/jov-2179)
25906816024  Fork PR Gate    codex/jov-2179
25906816169  CI              codex/jov-2179
25906801063  Auto-PR         codex/jov-2179
25906763897  Scope Judge     codex/jov-2199
25906763907  Workflow Lint   codex/jov-2199
25906762287  Workflow Lint   codex/jov-2199
```

Both `gh run cancel <id>` and `gh api -X POST /actions/runs/<id>/force-cancel` return HTTP 500 from GitHub. These runs are not consuming runners (they never picked up), but pollute `gh run list --status queued` output and may confuse the queue-depth measurements in R6 instrumentation.

## Why it matters

Low priority — not actively blocking anything. But future PR-yield + queue-wait measurements (R6) need to filter these out OR they're gone via GH-side garbage collection.

## Classification

Candidate (file GH support ticket if the runs persist past 90 days OR pollute R6 measurements). Pickup agent should first check whether the runs are still there via `gh run list --status queued`.

## Acceptance criteria

* Either: GH cleared the runs (queue list no longer shows them), OR
* GH support ticket filed and ticket ID recorded here, OR
* R6 instrumentation explicitly filters out runs older than 14 days

## Dependency

Soft-blocks R6 PR-yield/queue-wait measurement accuracy

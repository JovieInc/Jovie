# JOV-2123: Ops metrics data trust — add freshness indicators and per-source failure states

- **Status:** Todo
- **Priority:** 🟡 Medium
- **Assignee:** Unassigned
- **Team:** JOV
- **Project:** —
- **Labels:** area:ui, area:ops
- **URL:** https://linear.app/jovie/issue/JOV-2123/ops-metrics-data-trust-add-freshness-indicators-and-per-source-failure
- **Updated:** 2026-05-30T23:36:34.601Z

Ops metrics (MRR from Stripe, runway from Mercury, shipping velocity from GitHub/Linear) are fetched from external APIs. No freshness indicators currently exist.

## Problem

Per-source partial failure states are unspecified: what shows when Stripe is unavailable? Mercury? GitHub? No deployments? All blockers dismissed? Currently all failure modes look the same or silently show stale data.

Additionally, each KPI card is a dead end — there is no path from the metric to the owning system. A 100% reliability score has no link to Sentry or traces. An MRR figure has no link to the Stripe dashboard. A "Mercury unavailable" state shows no reason, owner, retry, or diagnostics link.

**Confirmed in code:** `HudDashboardClient.tsx` lines 533–591 (MRR/Runway) and 696–763 (Operations/Reliability) render as `ContentMetricCard` with no outbound links.

## Required

* Each metric card or strip item shows last-updated timestamp or staleness indicator.
* Per-source fetch failure shows a scoped error state (not a full-page error boundary).
* "No data" vs "data failed" are visually distinct.
* Each card links to its source system: MRR → Stripe dashboard, Runway → Mercury, Reliability → Sentry/incidents, Shipping velocity → GitHub/Linear.
* Unavailable source states ("Mercury unavailable," "Cannot calculate status") show: source name, reason if known, and a retry or link-to-configure path.

## Acceptance criteria

* MRR (Stripe) and runway (Mercury) show last-updated timestamps or staleness indicators.
* Per-source failures handled gracefully without full-page error boundary.
* Loading / partial / error / stale states specified and implemented for each source.
* Each KPI card has at least one outbound link to the owning system.
* "Unavailable" states show a reason and next step, not just a label.

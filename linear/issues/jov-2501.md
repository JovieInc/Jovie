# JOV-2501: Cost: share one Neon ephemeral branch across CI jobs instead of 10 per run

## Follow-up

Every CI run currently spins up **10 separate ephemeral Neon branches**, one per job that needs a DB:

```
neon-db (master pattern, already path-gated)
ci-mobile-overflow
ci-lighthouse-pr (public routes)
ci-lighthouse-dashboard-pr
ci-lighthouse-onboarding-pr
ci-lighthouse-admin-pr
ci-lighthouse-chat-pr
ci-a11y
ci-admin-smoke
ci-e2e-smoke
```

Each branch creates its own compute endpoint, runs migrations, runs the test job, then cleans up. The `neon-db` job already publishes `connection.json` as an artifact, but downstream jobs ignore it and create their own. This is what caused [JOV-2497](https://linear.app/jovie/issue/JOV-2497) (endpoint pool exhaustion) and contributes 5-10× to CI compute cost.

Fix options:

1. **Single shared branch per CI run (best)**. The `neon-db` job already creates and publishes `connection.json`. Refactor downstream jobs to consume that artifact instead of running their own `neon-create-branch-with-retry`. Branch count drops from 10 to 1 per run, endpoint pool stops blowing.
2. **Make Lighthouse + Mobile Overflow Guard DB-free**. Those jobs test visual layout, not DB queries. They could run against a Vercel preview build and never touch Neon. Cuts \~6 of the 10 branches.
3. **Tighter path-gating**. Apply the same `needs_db` gate the `neon-db` job uses to all downstream jobs — so a dependabot bump that touches only `package.json` does not create 10 branches.

Combine 1+2+3 for max savings.

## Why it matters

At 30-50 CI runs/day × 10 branches × 3-5 min of compute ≈ 100-200 CU-hours/mo on CI alone. Sharing one branch per run drops this 10× to 10-20 CU-hours/mo. Also eliminates the recurring Neon endpoint exhaustion that fails Lighthouse jobs and blocks the merge queue.

## Comments

### Tim White (2026-05-19)
**Drain-investigate session 2026-05-19 update — recommend rescope.**

Pulled this off the queue for the same /drain session that shipped JOV-2500. Got partway through scoping the shared-branch refactor and found enough complexity that I don't want to rush a CI workflow change.

What I confirmed reading `.github/workflows/ci.yml`:

1. **10 jobs use `neon-create-branch-with-retry`**: neon-db, ci-mobile-overflow, ci-lighthouse-pr, ci-lighthouse-dashboard-pr, ci-lighthouse-onboarding-pr, ci-lighthouse-admin-pr, ci-lighthouse-chat-pr, ci-a11y, ci-e2e-smoke, ci-admin-smoke.

2. **All non-neon-db jobs are path-gated** by `needs.ci-path-changes.outputs.run_neon == 'true'`. For non-DB PRs they fall back to `DATABASE_URL_MAIN`. So the 10-branch fan-out only fires when a PR touches DB-relevant files. Most PRs (UI, perf, docs) never spin up branches at all.

3. **Sharing one branch across audit jobs is harder than it looks**. Each audit job runs `drizzle:migrate:ci` against its own branch, then seeds test fixtures via `tests/seed-test-data.ts`. Drizzle migrations are idempotent against `__drizzle_migrations`, but a race window exists when N parallel jobs try to insert the same migration row. Seed functions (`ensureUserRecord`, `ensureCreatorProfileRecord`) are upsert-safe so that part is fine.

4. **The actual cost driver is the every-minute crons** (JOV-2500, shipped as #9329). CI ephemeral branches are a secondary cost contributor only when DB-relevant code changes. Most days the path-gate already prevents the fan-out.

**Recommendation: reframe this issue from "cost fix" to "operational hardening for JOV-2497".**

The endpoint exhaustion (JOV-2497) is real but the right fix is one of:

A) **Raise the Neon endpoint plan cap** — fastest, costs ~$5-10/mo more on the plan tier but eliminates the exhaustion class of failures entirely. Now that cost cap is $50/mo this is bounded.

B) **Migrate migrations into `neon-db` job** *before* the artifact upload, then refactor downstream jobs to consume the artifact without re-running migrations. This is the clean shared-branch refactor — bigger change, deserves its own session.

C) **Make Lighthouse + Mobile Overflow DB-free by switching to mocked SSR data**. Lighthouse currently audits real-feeling pages by seeding DB fixtures and starting next standalone; alternative is to render with a small JSON fixture set. Cleanest architecturally but requires a new SSR mocking layer.

I'd lean A as the immediate hardening + B as the right long-term refactor.

**Marking this for re-triage** rather than shipping a half-done refactor in this session. JOV-2500 (the real cost driver) is shipped via #9329 and that's the win for tonight.

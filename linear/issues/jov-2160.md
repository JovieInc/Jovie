## Follow-up

Add Playwright E2E coverage for the Clerk OAuth sign-up flow (Apple and Google) on `/start`, so the full `proposeNextStep(instant_access)` → inline `<SignUp />` → OAuth → `/sso-callback` → claim → `/onboarding/checkout` path runs end-to-end.

The [JOV-2147](https://linear.app/jovie/issue/JOV-2147/refactor-headernav-flyout-from-css-only-visibility-to-js-driven-hidden) bug (server-side redirect loop on `/start` after Apple sign-in) shipped past CI because Clerk's official Playwright helpers (`setupClerkTestingToken` + `createOrReuseTestUserSession`) only cover the email + OTP path. OAuth providers (`oauth_apple`, `oauth_google`) require either:

1. A stubbed OAuth adapter (mock the provider, drive the `/sso-callback` directly with a fixture session), or
2. Clerk's testing-mode OAuth tokens (if Clerk supports them on the test instance), or
3. Provider-test-account credentials wired into Doppler (heaviest option).

## Why it matters

We just shipped a P0 conversion-path bug that broke 100% of Apple-OAuth sign-ups on `/start`. No automated coverage caught it — the unit tests for the proxy rewrites passed, the components rendered, the claim API tests passed, but no test exercised the actual post-OAuth proxy interaction. Until OAuth is in CI, every email+OTP-tested change is a coin-flip for OAuth users.

## Triage question

Pickup agent must first judge approach: stubbed adapter (cheapest, most maintainable, doesn't catch real-provider regressions) vs. real test accounts (most realistic, fragile, secret-management overhead). Decision: pick one or close as not worth the maintenance cost.

## Acceptance criteria (if accepted, stubbed-adapter route)

* New spec `apps/web/tests/e2e/onboarding-start-oauth-claim.spec.ts`.
* Anonymous visitor lands on `/start`, types a message that triggers `propose_next_step` with `instant_access` (pin the LLM with a fixture or stub `/api/chat`).
* Stub the OAuth provider hop: directly drive `/sso-callback` with a fixture Clerk session that simulates a freshly-created user (mirrors what Apple OAuth produces).
* Assert: page URL lands at `/onboarding/checkout` within 8s, with no `ERR_TOO_MANY_REDIRECTS` and no `[proxy] Redirect loop circuit breaker triggered` Sentry breadcrumb.
* Assert: the `chat_conversations` row for the visitor's sessionId now has `user_id` set.
* Cleanup with `cleanup-e2e-users.ts --force` (per `.claude/rules/auth.md`).

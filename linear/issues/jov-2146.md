## Source

* Triggered by: [JOV-2145](https://linear.app/jovie/issue/JOV-2145/404-page-nav-renders-all-megamenu-flyouts-as-a-wall-of-unstyled-text) (404 nav megamenu rendered as wall of unstyled text)
* The current public-exhaustive Playwright lane checks status, text length, overflow, keyboard reach, and console errors — none catch missing-stylesheet bugs against visually-broken-but-semantically-valid markup. A targeted assertion was added in [JOV-2145](https://linear.app/jovie/issue/JOV-2145/404-page-nav-renders-all-megamenu-flyouts-as-a-wall-of-unstyled-text) (`assertMarketingHeaderFlyoutsHidden`), but that only catches the specific flyout pattern. Visual regression is the durable defense against the class of bug.

## Follow-up

Add Playwright `toHaveScreenshot` baselines on 6 critical public surfaces:

1. `/` (homepage)
2. `/pricing`
3. `/dev/root-not-found-probe` (renders root `app/not-found.tsx`)
4. `/legal/terms`
5. `/missing-qa-user` (renders `[username]/not-found.tsx`)
6. `/dualipa` (representative artist profile)

Mask dynamic regions (timestamps, counts, dynamic imagery, animated decorations). Run on PRs labeled `testing` only, not on every PR.

## Why it matters

Without screenshot baselines, the next CSS-cascade-order regression, design-token rename, or route-group orphan ships the same way [JOV-2145](https://linear.app/jovie/issue/JOV-2145/404-page-nav-renders-all-megamenu-flyouts-as-a-wall-of-unstyled-text) did. The targeted assertion added in [JOV-2145](https://linear.app/jovie/issue/JOV-2145/404-page-nav-renders-all-megamenu-flyouts-as-a-wall-of-unstyled-text) won't catch a different visual regression (e.g., footer mis-styled, dark-mode contrast bug, layout shift).

## Classification

Candidate follow-up — needs taste/UX review before implementation.

Pickup agent must first judge whether to implement, close, or split this.

## Acceptance criteria or triage question

If implementing:

* 6 baseline screenshots committed (desktop + mobile = 12 images)
* Pixel diff threshold tuned to avoid flake (anti-aliasing, font hinting drift)
* New test fixture in `apps/web/tests/e2e/visual-regression.spec.ts`
* Wired to `testing` label in `.github/workflows/ci.yml`
* Documentation in `apps/web/tests/TESTING.md` for updating baselines

Triage question: is the operational cost of maintaining 12 baseline images (intentional design refreshes require a baseline update flow) worth the regression coverage?

## Follow-up

The `/app/admin/ops` browser smoke in local dev still logs `Error loading Mercury metrics` with `Mercury API error (404): {"errors":{"notFound":["We couldn’t find the data associated with your request. Please contact help@mercury.com"]}}`. The route falls back enough to render, but the server log remains noisy and the default status uses unavailable Mercury data.

## Why it matters

Ops should be calm by default. A recurring server error in the command center makes it harder to distinguish expected unavailable data from real regressions.

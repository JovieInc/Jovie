## Follow-up

The approve endpoint (`POST /api/connectors/suggested-actions/[id]/approve`) performs two sequential writes: CAS update on `suggested_actions` (pending → accepted) and insert into `workflow_runs`. These cannot be wrapped in a transaction per `.claude/rules/db.md`.

If the CAS update succeeds but the `workflow_runs` insert fails, the action is left in `accepted` status with no corresponding `workflow_runs` row. The endpoint returns 500 so the client knows the operation failed. On retry, the CAS misses (409 "already-decided") because the row is now accepted.

At closed-beta DJ scale (single design-partner) this surfaces quickly and can be recovered manually. However, once the beta is rolled out more broadly, a compensating recovery scan should exist.

The recovery scan would query for `suggested_actions` rows with `status = 'accepted'` that have no corresponding `workflow_runs` row (LEFT JOIN + IS NULL), then re-insert the missing runs. This could run as part of the existing `process-workflow-runs` cron tick (add a pre-scan step) or as a separate nightly reconciliation job.

Flagged by CodeRabbit in PR #8813 (comment id:3252955992). Documented in the JSDoc of the approve route.

## Why it matters

Without this, a user who approves a calendar event but hits a transient DB error on the insert will be silently stranded — their action is "accepted" but the calendar event is never created.

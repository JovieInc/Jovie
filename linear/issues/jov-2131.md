## Source

## Follow-up

PR #8497 introduced `process.env.NEXT_PUBLIC_CLERK_OAUTH_<PROVIDER>_ENABLED === '1'` static lookups in `apps/web/lib/auth/oauth-providers.ts` to fix PR #8458's broken bracket-notation env-var gate. The static-reference approach is what Next.js / webpack DefinePlugin documentation says is required for `NEXT_PUBLIC_*` inlining at build time.

End-to-end verification on 2026-05-10:

1. Env vars added to Vercel production:
   * `NEXT_PUBLIC_CLERK_OAUTH_APPLE_ENABLED="1"`
   * `NEXT_PUBLIC_CLERK_OAUTH_GOOGLE_ENABLED="1"`
2. Confirmed via `vercel env pull --environment=production` — both values present.
3. PR #8497 merged (commit fd17329) and deployed cleanly through the canary + production promote pipeline.
4. Production health check `/api/health/build-info` confirmed `commitSha: fd17329` was live.
5. **Browser fiber inspection** of `<SignIn>` component's `appearance.elements` prop still showed all 12 `socialButtonsBlockButton__*` hidden keys, including apple and google. The function was still returning false for every provider.

Diagnosis ran out of budget mid-incident — production auth UI was empty for all users so we pivoted to PR #8499 (hardcoded allowlist) which works.

## Hypothesized causes (none confirmed)

1. Turborepo cache reused a previously-compiled `oauth-providers.ts` chunk from before env vars were set; the new source-level change wasn't enough to invalidate the cache key.
2. Vercel build env vars are scoped differently than runtime env vars and `NEXT_PUBLIC_*` weren't exposed to the Turbopack/webpack DefinePlugin pass.
3. Subtle bundler interaction where switch-case on a string parameter is not statically analyzable enough for DefinePlugin to substitute correctly.

## Why it matters

Several existing repo conventions rely on `NEXT_PUBLIC_*` build-time inlining (CSP nonces, feature flags, public Clerk publishable key). If those inlining assumptions are broken in this repo's specific Vercel + Turborepo setup, other features could silently fall back to disabled-by-default behavior the next time someone adds one.

The hardcoded allowlist in `apps/web/lib/auth/oauth-providers.ts` is the live workaround. Once the root cause is understood, we may want to revisit (or document the limitation explicitly so future PRs don't reintroduce env-var gates that look correct but never fire).

## Acceptance criteria or triage question

* Reproduce the failure on a preview deploy: add a throwaway `NEXT_PUBLIC_DEBUG_FLAG=hello` env var, reference it in a `'use client'` component via `process.env.NEXT_PUBLIC_DEBUG_FLAG`, deploy, and check whether the literal "hello" appears in the built JS bundle.
* If reproduces: file a focused bug against Turborepo or Vercel with the smallest repro.
* If does not reproduce: examine the specific build that produced commit fd17329 — was Turbo cache hit on the oauth-providers task? Was the env var actually present in that build's runtime environment?
* Decide whether to keep the hardcoded allowlist as the canonical pattern or restore env-var gating once the cause is understood.

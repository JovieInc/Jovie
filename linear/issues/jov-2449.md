## Follow-up

[JOV-2446](https://linear.app/jovie/issue/JOV-2446/sso-only-auth-layer-a-ui-canary-clerk-config-audit-cron) ships Layer A (unauthenticated UI smoke) but explicitly defers Layer B (authenticated-session canary using Clerk Backend SDK `signInTokens.createSignInToken` against a pre-provisioned synthetic prod user).

The deferral reason: a permanent canary user in production Clerk creates a real long-term liability — exclusion from marketing emails, analytics rollups, MRR calculations, entitlement audits, RLS test fixtures — plus a standing credential that can mint sessions for anyone via the same Backend SDK call. Sentry + RUM already capture middleware/protected-route regressions.

**Acceptance gate:** Only open this for implementation if Layer A + Sentry + RUM together MISS a real production session/middleware regression for ≥15 minutes. The incident must be specifically named in the PR description ("opened in response to <Linear incident issue ID>"). Without that gate, this issue should stay in Backlog or be closed.

## Why it matters (when triggered)

Layer A validates the auth UI surface but not the authenticated path. If Clerk session validation, middleware auth chain (`apps/web/proxy.ts`), protected-route serving, DB connectivity for an authed user, or entitlement resolution silently regresses, Sentry and RUM should fire — but if they don't, we need a second layer.

## Implementation reference (from [JOV-2446](https://linear.app/jovie/issue/JOV-2446/sso-only-auth-layer-a-ui-canary-clerk-config-audit-cron) plan / autoplan review)

* Mechanism: `clerkClient.signInTokens.createSignInToken({ userId, expiresInSeconds: 60 })`; navigate Playwright to the returned `url` field (not a hand-rolled `?__clerk_ticket=...`).
* Synthetic user: pre-provisioned in prod Clerk with `publicMetadata: { role: 'canary', excludeFromAnalytics: true }`. Disable MFA on the user. Store the `userId` as Doppler secret `CLERK_PROD_CANARY_USER_ID` in `jovie-web/prd`.
* Hard requirement: explicit exclusion filters in every marketing send path, analytics aggregation, MRR query, RLS test fixture, and admin user-listing query before the user goes live. File per-system tracking Linear issues.
* New spec: `apps/web/tests/e2e/synthetic-authenticated.spec.ts` + helper `apps/web/tests/e2e/utils/production-session-canary.ts`.
* Workflow: extend `synthetic-monitoring.yml` with a new step that runs the spec.

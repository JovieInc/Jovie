# JOV-2259: fix(tasks): TasksUpgradeInterstitial card uses semi-transparent app-content-surface (potential elevation bug in dark mode)

## Source

Discovered during ad-hoc UI ideation audit on 2026-05-15 (parent: ad-hoc swarm session — no parent issue).

## Follow-up

The `TasksUpgradeInterstitial` non-compact variant renders with `bg-[color-mix(in_oklab,var(--linear-app-content-surface)_97%,transparent)]` — essentially a 97% opaque version of the shell canvas background on a card that floats on top of the shell canvas:

File: `apps/web/components/features/dashboard/tasks/TasksUpgradeInterstitial.tsx:36`

```tsx
'max-w-xl rounded-[24px] border border-[color-mix(in_oklab,var(--linear-app-shell-border)_72%,transparent)] bg-[color-mix(in_oklab,var(--linear-app-content-surface)_97%,transparent)] px-8 py-14'
```

Per `.claude/rules/ui.md` banned patterns: "bg-(--linear-app-content-surface) on card-like elements inside the shell" and "bg-surface-0/XX (semi-transparent) — use solid bg-surface-0 instead". A semi-transparent content-surface background on an upgrade card is visually nearly identical to the shell, making the card invisible.

The upgrade interstitial is a high-stakes conversion surface (visible to all non-Pro users navigating to /app/dashboard/tasks).

Fix: Change to `bg-surface-1` with the existing border, OR use `bg-surface-0` for a recessed-well look. Remove the `color-mix` transparency.

## Why it matters

This is the primary paywall screen for the Tasks feature — used to convert free users. If the card is nearly invisible against the dark shell canvas, the upgrade CTA loses impact.

## Classification

Required (hard invariant — banned semi-transparent surface pattern; conversion surface)

## Acceptance criteria

* TasksUpgradeInterstitial uses solid `bg-surface-1` or `bg-surface-0` (no semi-transparent color-mix)
* Card is visually distinct from shell canvas in both light and dark mode
* Upgrade button remains the primary visual focal point

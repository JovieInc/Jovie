## Source

* Parent: [JOV-2148](https://linear.app/jovie/issue/JOV-2148/hardened-coverage-for-human-eye-only-ui-inconsistencies)

## Follow-up

Add a custom ESLint rule that flags string literals passed to known label-positioned props (`Button` children, `Badge` children, `TabsTrigger` children, `CardTitle` children) where the casing doesn't match the per-context rule from `DESIGN.md`:

* Title Case for labels/buttons/badges/headings
* Sentence case for body/tooltips/toasts

## Why it matters

Gotcha class #9 — inconsistent casing across the same screen looks broken to users. `.claude/rules/ui.md` already has the rule but enforcement is manual.

## Classification

Candidate (the rule is heuristic and may produce false positives)

## Triage question

Is casing-by-context detectable accurately enough by static analysis, or is this better handled at the design-system component level (e.g. `<Button>` lowercases or uppercases its own children)?

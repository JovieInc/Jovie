# JOV-2341: [Infra/Cost] Serialize concurrent CI runs that create Neon branches (avoid concurrent endpoint cap)

- **Status:** Backlog
- **Priority:** 🟡 Medium
- **Assignee:** Unassigned
- **Team:** JOV
- **Project:** —
- **Labels:** codex
- **URL:** https://linear.app/jovie/issue/JOV-2341/infracost-serialize-concurrent-ci-runs-that-create-neon-branches-avoid
- **Updated:** 2026-05-31T04:58:17.519Z

## Why

When multiple CI reruns trigger simultaneously (e.g., manual rerun on 3 PRs, or autopilot batch), each creates a Neon ephemeral branch + compute endpoint at the same time. On Launch plan there's a concurrent endpoint cap that returns HTTP 402 and fails the run with "concurrent endpoint limit exceeded". Saw this in [JOV-2303](https://linear.app/jovie/issue/JOV-2303/securityinfra-ci-neon-ephemeral-db-credentials-failing-blocks-open-prs) follow-up runs.

Beyond hitting the cap, even when under it, multiple concurrent runs multiply the per-run compute cost.

## What to do

Add a workflow-level `concurrency` group to `.github/workflows/ci.yml` that limits Neon-branch-creating CI runs to N at a time (e.g., 3). Pattern:

```yaml
concurrency:
  group: ci-neon-branches
  cancel-in-progress: false  # queue, don't kill in-flight
```

Or use `actions/queue-action` if the existing concurrency primitive isn't enough. Be careful: don't queue jobs that don't need Neon (per \[Infra/Cost\] CI: skip Neon ephemeral branch for PRs that don't touch DB-affecting code), since serializing the whole pipeline hurts throughput.

## Acceptance

- [ ] At most N concurrent CI runs hold a Neon ephemeral endpoint
- [ ] HTTP 402 "concurrent endpoint limit" errors stop appearing in CI logs
- [ ] Total CI wall-clock doesn't increase for typical 1-PR workflows (only multi-PR bursts queue)

## Dependency

Best to land \[Infra/Cost\] CI: skip Neon ephemeral branch for PRs that don't touch DB-affecting code first, so the queue is mostly DB-relevant PRs.

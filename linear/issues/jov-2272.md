## Follow-up

While shipping [JOV-2180](https://linear.app/jovie/issue/JOV-2180/epic-billing-ux-hardening-cancel-period-end-recon-scope-trial) subset #1 (`/api/stripe/cancel` now schedules cancellation at period end), we discovered that the `customer.subscription.updated` webhook handler (`apps/web/lib/stripe/webhooks/handlers/subscription-handler.ts`) and the `users` schema (`apps/web/lib/db/schema/auth.ts`) do NOT persist:

* `cancel_at_period_end` (bool)
* `cancel_at` or `current_period_end` (timestamp)

Without these fields, the billing UI cannot render a persistent "Scheduled to cancel on <DATE>" banner after a page refresh. Subset #1 degraded to toast-only feedback on the cancel action — the user only sees the cancel-on date in the success toast, and there is no visual indicator on the billing page after that.

We also need to add a "Reactivate" CTA (which calls `subscriptions.update(id, { cancel_at_period_end: false })`) so users can undo the scheduled cancellation before the period ends.

## Why it matters

* Without a persistent banner, users have no in-product confirmation that their subscription will end. This will drive support tickets ("did my cancel work?").
* Without a Reactivate path, users who change their mind have to contact support or wait until the cancellation lands and re-subscribe (worse UX, lost revenue).
* Stripe's reactivation API is free and instant; building this is cheap.

## Classification

Required (to complete [JOV-2180](https://linear.app/jovie/issue/JOV-2180/epic-billing-ux-hardening-cancel-period-end-recon-scope-trial) acceptance criteria item: "Cancel UI shows 'Your Pro access continues until {periodEnd}' + Undo affordance")

## Acceptance criteria

- [ ] Add two columns to `users`: `cancel_at_period_end boolean default false`, `cancel_at timestamp null` (or equivalent on the existing billing fields)
- [ ] Migration is additive-only per `.claude/rules/db.md`
- [ ] `SubscriptionHandler.handleUpdated` writes both fields when Stripe sends `cancel_at_period_end: true|false`
- [ ] `SubscriptionHandler.handleDeleted` clears both fields
- [ ] `getUserBillingInfo` / billing status query exposes both fields to the client
- [ ] `BillingDashboard` renders a persistent banner: "Scheduled to cancel on <DATE>. Reactivate?" when `cancelAtPeriodEnd` is true
- [ ] "Reactivate" button calls a new `/api/stripe/reactivate` endpoint that runs `subscriptions.update(id, { cancel_at_period_end: false })`
- [ ] Regression test: webhook handler persists both fields on `customer.subscription.updated`
- [ ] Regression test: reactivate route calls Stripe with `cancel_at_period_end: false`

## Dependency

Depends on [JOV-2180](https://linear.app/jovie/issue/JOV-2180/epic-billing-ux-hardening-cancel-period-end-recon-scope-trial) subset #1 (cancel-period-end) landing first so the cancel route returns a valid `cancelAt` to confirm against.

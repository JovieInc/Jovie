# JOV-1999: feat(agentos): wire Sentry + CI + Linear signals into ops_events

- **Project:** —

- **Updated:** 2026-05-31T04:59:09.126Z

Wire the first emitters to populate `ops_events`. Start with 3 signal sources:

1. **Sentry spike** — when a new error group exceeds threshold (N events in M minutes), insert ops_event with source='sentry', severity based on Sentry level
2. **Failed CI** — GitHub Actions webhook on `workflow_run.completed` with conclusion='failure', insert ops_event with source='ci'
3. **Stale PR** — cron or webhook: PR open >72h with no activity, insert ops_event with source='linear'

## Rules from infra guardrails

* Use webhooks (Sentry + GitHub webhook) — not polling
* If adding a new cron, justify it in the PR and add to `docs/CRON_REGISTRY.md`
* All external HTTP must use bounded fetch (timeout + retry wrapper from `@/lib/security.md`)
* Insert via `db.insert(opsEvents).values([...])` — no `db.transaction()`

## Acceptance criteria

* Sentry webhook handler at `/api/webhooks/sentry/ops-events`
* GitHub Actions webhook handler at `/api/webhooks/github/ci-failure`
* Stale PR detection (cron or webhook — justify choice in PR)
* `ops_events` rows inserted with correct severity/impact/source mapping
* Cost Impact section in PR description

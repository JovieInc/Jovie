## Follow-up

Two taste-level improvements to `scripts/check-conflict-markers.sh` that were deferred from the initial implementation:

1. **Silent happy path** — the script exits 0 with no output when no markers are found. On slow machines or in CI, developers see nothing and may wonder if the check ran at all. A brief `echo "✓ No conflict markers found"` (or the equivalent without emoji per repo rules) would confirm the hook fired.
2. **File names missing from error output** — when markers are detected, the current error message shows the raw `git diff --cached --check` lines but not a clean list of which files are affected. Adding a `git diff --cached --name-only | xargs -I{} echo "  {}:"` pass or filtering the check output to surface file paths would make it faster to find and fix the conflicted file.

## Why it matters

Agents committing after a rebase will see the error but may waste time grepping for the conflicted file. Surface the file name directly to reduce the fix cycle from \~30s to \~5s.

## Classification

Candidate

## Acceptance criteria or triage question

Should we add a success confirmation line (non-empty stdout on clean pass) and should the error output list affected file paths explicitly? Or is the current silent-success + raw check output sufficient for the intended agent-facing use case?

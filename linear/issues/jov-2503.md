## Source

* Current issue: ad-hoc, discovered during JOV-2500 implementation 2026-05-19

* Source branch/session: fix/cron-reduce-frequency-neon-cost

## Follow-up

`vercel project inspect jovie` shows the project Root Directory is `apps/web/`, which means Vercel reads `apps/web/vercel.json` exclusively. The root-level `vercel.json` is dead code that contains a near-duplicate of the live config plus one extra cron (`public-profile-canary`) that does not actually run because Vercel never sees this file.

The stale file is dangerous: it tricked one investigation into thinking the cron cadence change had been applied (it had — to the wrong file). `docs/CRON_REGISTRY.md` was also out of date in the opposite direction, claiming the root file was source of truth.

## Fix

1. Delete repo-root `vercel.json`
2. Verify `vercel deploy` (preview) still picks up `apps/web/vercel.json` correctly
3. Add a `# This file is intentionally absent — Vercel reads apps/web/vercel.json` note to AGENTS.md if useful, or just remove it cleanly
4. Confirm `public-profile-canary` cron is still scheduled (it appears in root vercel.json but NOT in apps/web/vercel.json — that means it has not been running). If it was supposed to be running, add it to `apps/web/vercel.json`.

## Why it matters

Future agents will keep editing the wrong file. The cost is bounded (just slow next time) but the risk is real (production cron change that silently does nothing). Already cost me one investigation cycle today.

Also: `public-profile-canary` may have been broken since the file split — worth verifying.

## Classification

Candidate (correctness + clarity, not urgent)

Pickup agent must first judge whether to implement, close, or split this.

## Acceptance criteria or triage question

- [ ] If yes, add it to `apps/web/vercel.json` before deleting root `vercel.json`
- [ ] If no, that is itself a bug to fix
- [ ] Delete root `vercel.json` and verify next deploy still works

# JOV-2497: CI: Neon ephemeral endpoint pool exhaustion blocking Lighthouse + A11y

## Follow-up

Neon project `curly-dew-35158967` is hitting the 'concurrently active endpoints' cap. When CI fans out across the open PR queue, every job (Lighthouse public/admin/chat/onboarding, A11y axe, Mobile Overflow Guard) creates its own ephemeral branch and connects to the compute endpoint. The pool saturates, then every subsequent run loops through 12 retries (5s each) and dies with `You have exceeded the limit of concurrently active endpoints`.

Observed during this drain:

* 3 consecutive `main` CI failures across \~9h (A11y job exhausted)
* 8+ open PRs failing Lighthouse (public routes PR) with the identical Neon log signature
* All Mobile Overflow Guard widths failing on the same PRs with the same root cause

The `neon-ephemeral-branch-cleanup.yml` workflow only triggers on PR close — it does not relieve pressure while the queue is hot. Branches do auto-TTL at 4h, so the pool naturally drains, but a single rebase-the-world pass can blow the cap.

Likely fixes worth scoping:

1. Raise the Neon plan endpoint cap (paid plan increase) — fastest
2. Concurrency cap on CI: limit how many PRs' Lighthouse + A11y + Mobile Overflow jobs can run simultaneously (workflow-level `concurrency.group` + `cancel-in-progress` is partially in place, but cross-PR concurrency isn't capped)
3. Share a single Neon branch per `merge_group` instead of per PR per job
4. Add a scheduled `neon-ephemeral-branch-cleanup` (every 30m) that suspends or deletes endpoints older than 1h with no recent traffic
5. Make Lighthouse + Mobile Overflow not require DB (run against a static preview build) so they don't compete for Neon at all

## Why it matters

Drain operations cannot reach zero while this is hot — the entire PR queue cascades into Neon-induced CI failures even on tiny dependabot bumps. It also masks real regressions (looks like Lighthouse failed → may actually be infra).

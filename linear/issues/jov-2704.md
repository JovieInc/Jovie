# JOV-2704: MEMORY: ship v0 studio-session memory loop

- **Status:** Todo
- **Priority:** 🔴 Urgent
- **Assignee:** Tim White
- **Team:** JOV
- **Project:** Memory Core / Content Engine
- **Labels:** area:memory, codex, type:feature
- **URL:** https://linear.app/jovie/issue/JOV-2704/memory-ship-v0-studio-session-memory-loop
- **Updated:** 2026-05-31T08:09:17.548Z

## Goal

Ship the first end-to-end Jovie Memory Core loop behind default-off flags:

> A creator tags a person in a photo. Jovie creates/enriches the person entity, correlates nearby Gmail/Calendar context, creates a studio-session event, links person/location/song/assets, and proposes an approval-gated content opportunity with source evidence.

## Architectural decision

This project is customer-facing product memory. It does not replace the existing internal AgentOS WDK proof.

* Product memory durable workflows: Trigger.dev behind a `WorkflowRunner` abstraction.
* Internal AgentOS control plane: WDK/Vercel Workflow remains scoped to existing AgentOS/Hermes/Ruflo work until a separate ADR changes it.
* Canonical memory: Neon Postgres + Drizzle. External memory products may be used only as secondary indexes/adapters.
* Agent harness: `AgentHarness` interface with OpenAI Agents SDK adapter as the target; use the current AI SDK Gateway extraction path as a compatibility fallback during rollout.
* Connectors: native Google first; Composio only behind `ConnectorProvider` and only for read-only/low-risk POCs.

## Current repo facts to respect

* Web app is in `apps/web`, Drizzle schema is `apps/web/lib/db/schema/index.ts`, migrations are `apps/web/drizzle/migrations`.
* Database is Neon Postgres through `@neondatabase/serverless`; do not add a parallel DB client.
* Existing connector tables already exist: `connector_accounts`, `connector_sync_states`, `external_objects`, `context_facts`, `agent_runs`, `suggested_actions`, `workflow_runs`.
* Existing Google OAuth callback writes Gmail and Calendar accounts, but connector enums currently need cleanup.
* Existing chat pipeline is provider-neutral around AI SDK Gateway. Do not rewrite chat wholesale.

## Acceptance

* Child issues are implemented as isolated PRs/worktrees with narrow file scopes.
* v0 is flag-gated and safe to deploy with no social posting/write actions enabled by default.
* Every memory fact has provenance, confidence, owner/user scope, and evidence links.
* Demo path works on seeded/dev data with at least one tagged person/photo, Gmail/Calendar context, a studio-session event, and one content opportunity.
* Deferred work remains open as explicit Linear issues rather than hidden TODOs.

## Non-goals

* Full Google Photos/iCloud sync.
* Voice memo/audio parsing and TikTok video generation.
* Social publishing/write scopes.
* Broad pgvector recall before Memory Core v0 writes clean objects.
* Making Composio canonical.
* Making Anthropic/OpenAI hosted memory canonical.

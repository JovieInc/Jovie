# JOV-1845: Persist TCPA consent (text hash + version) on legacy SMS subscribe path

- **Status:** Backlog
- **Priority:** 🟠 High
- **Assignee:** Tim White
- **Team:** JOV
- **Project:** Launch: YC S26
- **Labels:** type:feature
- **URL:** https://linear.app/jovie/issue/JOV-1845/persist-tcpa-consent-text-hash-version-on-legacy-sms-subscribe-path
- **Updated:** 2026-05-31T02:18:21.243Z

## Problem

Surfaced by the Wave B1 ([JOV-1840](https://linear.app/jovie/issue/JOV-1840/phase-2-public-alert-landing-page-with-source-link-attribution)) adversarial review. The legacy form-input SMS subscribe path goes through `apps/web/lib/notifications/domain.ts:518-562` (`buildSubscriptionValues`) which writes a row to `notification_subscriptions` but does **not** populate `smsConsentTextHash` / `smsConsentVersion`, even though the schema has those columns and the inbound STOP webhook already writes them.

## Why it matters

Affects every fan who subscribes via SMS through the legacy form path, including the existing inline profile CTA and the new Wave B1 alert landing page. If a TCPA dispute arrives we have IP + source string + timestamp but no proof of which exact consent text the user saw — we'd be defenseless about the specific copy.

The Phase 1 ([JOV-1834](https://linear.app/jovie/issue/JOV-1834/native-sms-subscribe-handoff-phase-1-intent-api-webhook-frontend-cta)) native SMS handoff captures consent on the `smsSubscribeIntents` row, but that's a different surface. Anyone going through `POST /api/notifications/subscribe` with `channel='sms'` skips the consent ledger.

## Scope

1. Hash the canonical SMS consent text at server-receive time using the existing `getSmsConsentTextHash()` helper in `apps/web/lib/notifications/sms-consent.ts`.
2. Persist `smsConsentAt`, `smsConsentTextHash`, `smsConsentVersion` on the `notification_subscriptions` row in `buildSubscriptionValues` for `channel === 'sms'`.
3. Apply the same atomicity invariant the schema already enforces (all-or-none).
4. Backfill best-effort for existing rows using the timestamp + the v1 consent text/hash where unambiguous.

## Acceptance

- [ ] New SMS subscribers via the legacy path get all three consent fields populated.
- [ ] Atomic check constraint (if any) still passes.
- [ ] No regression to email-channel subscriptions (consent fields stay NULL).
- [ ] Backfill script logged, run in staging first.

## References

* `apps/web/lib/notifications/domain.ts:518-562` — current `buildSubscriptionValues`
* `apps/web/lib/db/schema/analytics.ts` — `notification_subscriptions.smsConsentAt/TextHash/Version`
* `apps/web/lib/notifications/sms-consent-shared.ts` — canonical text + version
* `apps/web/lib/notifications/sms-consent.ts` — `getSmsConsentTextHash()`

## Out of scope

* Migrating to the native handoff for landing pages (separate UX call)
* The same gap on `audience_actions` opt-in events (file separately if needed)

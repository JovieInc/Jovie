## Source

* Current issue: ad-hoc (CI/Jovie-bot autoplan, 2026-05-18)
* Source plan: R6 (new, surfaced by both CEO and Eng phases)

## Follow-up

Three weekly cron measurements wired into the HUD:

1. **PR-yield**: `(merged_agent_PRs / opened_agent_PRs)` over rolling 30 days, filtered by branch prefix (`codex/**`, `claude/**`, `*/jov-**`). Also `% of merged agent PRs with human repair commits` (any commit by non-`jovie-bot[bot]` author after the agent's first commit). New script: `scripts/hud/pr-yield.ts`.
2. **Agent intelligence $/merged-PR**: estimate from OpenRouter/Anthropic usage logs (Doppler scope `jovie-web/dev_personal` LINEAR_API_KEY, but usage logs may be elsewhere — verify in Doppler) divided by merged agent PR count. Output weekly. New script: `scripts/hud/agent-cost.ts`.
3. **Bot review latency**: for each closed PR last 30 days, compute `first_bot_review_submittedAt - PR.createdAt` for `coderabbitai[bot]` and `greptile-apps[bot]`. Output p50/p95. New script: `scripts/hud/bot-review-latency.ts`.

## Why it matters

Both review phases independently flagged these as the actual axis to optimize. CI minutes are NOT the dominant cost; agent intelligence + PR yield + bot review latency are. Without these metrics, any provider-change decision (R3 Stage B/C) is gating on the wrong bottleneck.

Re-evaluate triggers (from plan):

* PR-yield < 70% over 30 days → narrow agent scope or improve task decomposition
* Agent $/merged-PR > $5 → route more to smaller models or cache more context
* Bot review p95 > CI p95 → all R3 work is wrong-axis; pause it

## Follow-up

The email signature builder (`apps/web/lib/email-signature/build-signature.ts`) already accepts a `hideJovieBranding` flag, and `buildSignatureInputFromProfile` in `profile-input.ts` passes it through. However, no caller currently sources the flag from a per-profile setting — there is no `hideJovieBranding` (or "remove Jovie branding") field on `creator_profiles` today and no Pro entitlement gates the footer.

When that toggle ships (either as a `creatorProfiles.settings` key, a top-level column, or an entitlements flag), plumb it into both call sites:

* `apps/web/components/features/admin/admin-creator-profiles/AdminCreatorProfilesUnified.tsx` (admin "Email signature" action — reads `signatureProfile`)
* `apps/web/components/features/admin/admin-creator-profiles/hooks/useContextMenuItems.tsx` (admin "Copy email signature" context menu item — reads `profile`)
* `apps/web/components/features/dashboard/organisms/profile-contact-sidebar/ProfileContactSidebar.tsx` (dashboard overflow menu — reads `previewData`)

## Why it matters

The branding footer is a deliberate acquisition loop. We want it on by default, but Pro/founding/trial users who opt out via the (future) "remove Jovie branding" setting must see the footer suppressed in their signature — otherwise the toggle is silently ignored.

## Acceptance criteria or triage question

Triage question: does Jovie have a per-profile "remove Jovie branding" setting today (or planned)? If yes, source the flag here. If no, close this — the v1 default (always include the footer) stays correct until that setting exists.

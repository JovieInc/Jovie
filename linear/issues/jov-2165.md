## Follow-up

Migration 0046 (`social_links_creator_platform_url_unique`) creates the partial unique index with a blocking `CREATE UNIQUE INDEX` because it has to run inside Drizzle's migrator transaction. CodeRabbit flagged this as a production-scale concern.

If `social_links` grows past roughly \~1M rows, the blocking index build can hold a SHARE lock long enough to cause user-visible write latency. The pickup path is:

1. Run `CREATE UNIQUE INDEX CONCURRENTLY social_links_creator_platform_url_unique_v2 ON social_links USING btree (creator_profile_id, platform, normalize_social_url(url)) WHERE is_active = true AND state = 'active';` outside any transaction (`psql` directly against the production replica, not via Drizzle's migrate runner).
2. After it succeeds, rename the old index out, rename the new index in, drop the old.
3. Update `lib/db/schema/links.ts` to match (no functional schema change — same expression, just a fresh build).

## Why it matters

Prevents future migration regressions from causing write-latency spikes on a hot table when the system scales.

## Acceptance criteria or triage question

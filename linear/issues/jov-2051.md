## Follow-up

The merged `desktop-release.yml` on `main` is currently `workflow_dispatch`-only. Re-introduce the path-filtered push trigger from PR #8353 so signed releases publish automatically when the binary actually changes:

```yaml
on:
  push:
    branches: [main]
    paths:
      - "apps/desktop/**"
      - "VERSION"
      - ".github/workflows/desktop-release.yml"
  workflow_dispatch:
    ...
```

Important: keep the **App Store Connect API key** secret set that's already wired (`CSC_LINK`, `CSC_KEY_PASSWORD`, `APPLE_API_KEY`, `APPLE_API_KEY_ID`, `APPLE_API_ISSUER`, `APPLE_TEAM_ID`). Do **not** revert to PR #8353's app-specific-password flow (`MAC_CERTIFICATE_BASE64` + `APPLE_APP_SPECIFIC_PASSWORD`) â€” that path is now retired.

Also as part of this PR:

* Close PR #8353 as superseded (draft, conflicting, wrong secret set).
* Default `environment` for push-triggered runs to `production`; preserve `staging` as a `workflow_dispatch` input.
* Concurrency group: `desktop-release-${{ github.event.inputs.environment || 'production' }}`, `cancel-in-progress: false`.

## Why it matters

Tim shouldn't have to manually click a button to ship a desktop update when desktop code or VERSION changes. Today, even after a meaningful native/shell change, a release only happens via `gh workflow run`. With the path filter in place: every change that actually warrants a new binary auto-publishes a signed + notarized DMG, and electron-updater on installed apps picks it up.

Note for reviewer: the desktop binary does **not** need to rebuild on every web push. The Electron shell loads `jov.ie` live in a webview, so web content updates propagate to installed apps without a new release. The path filter (`apps/desktop/**`, `VERSION`, the workflow file) is the correct heuristic.

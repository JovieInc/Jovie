## Follow-up

`apps/web/proxy.ts` (catch block around line 1231) currently returns the raw exception `name` and `message` in the 502 JSON response body:

```ts
return NextResponse.json(
  {
    error: 'Clerk proxy error',
    code: errName,
    hint: errMessage,
  },
  { status: 502 }
);
```

This was intentionally added during the Apple Sign In production incident on 2026-05-10 to make the next proxy failure debuggable from the browser. It paid off — the `hint` was what surfaced "URL is malformed `/v1/oauth_callback?...`" and led directly to the relative-Location fix in PR #8503.

Now that the OAuth flow is stable and verified end-to-end (Apple Sign In + Hide My Email working in production on commit c4af74d), the `hint` field should be removed so we don't leak runtime exception messages to the browser.

Keep:

* `console.error('[clerk-proxy] fetch failed:', { name, message, cause, method, path })` — structured server-side logging for Sentry
* `code: errName` in the 502 body — generic enough not to leak useful internals

Remove:

* `hint: errMessage` from the 502 body

## Why it matters

Defense in depth. Exception messages can contain FAPI hosts, internal paths, or runtime details that don't belong in untrusted client responses. The Codex review on PR #8471 flagged this as a "consider after incident" cleanup.

## Acceptance criteria or triage question

If implemented: delete the `hint:` line in `apps/web/proxy.ts`'s `clerk-proxy` catch block, keep the structured `console.error` logging untouched. No test changes needed.

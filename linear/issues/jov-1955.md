## Follow-up

The pasted memo that drove the find-skills install also recommends building Jovie-native artist skills as a product surface, distinct from the engineering-time `find-skills`/`vercel-labs/skills` ecosystem. Candidate skill set from the memo:

* `jovie-release-planner`
* `jovie-playlist-pitch`
* `jovie-artist-profile-builder`
* `jovie-fan-notification-segmenter`
* `jovie-social-content-remixer`
* `jovie-metadata-checker`
* `jovie-show-promo`
* `jovie-drop-followup`

These would map directly to Jovie positioning ("plan releases, create assets, pitch playlists, promote drops from one AI workspace") and let us tell a "specialized artist skills for every part of a release" product story.

## Why it matters

* Maps current Jovie marketing language onto a concrete capability surface artists can recognize.
* Differentiates from generic bio-link / "AI assistant" tools by codifying repeatable artist workflows.
* Contains the surface area: artists never browse the open ecosystem; they only see Jovie-blessed skills.
* Sets up future agent automation (artist HUD, autopilot) on a stable named-workflow contract.

Candidate

## Acceptance criteria or triage question

Decision question for pickup agent: do we build these as

1. Anthropic-format Agent Skills (filesystem `SKILL.md`s) consumed by an internal Jovie agent runtime,
2. Jovie product features in the dashboard (no Agent Skill packaging at all), or
3. A hybrid where Jovie product UIs invoke a Jovie agent that loads the skill bundle internally?

Pickup agent must first judge whether to implement, close, or split this.

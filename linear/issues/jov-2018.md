# JOV-2018: EPIC: Public Profile Hardening — Mobile-Native, High-Traffic Reliability

## Mission

Harden the public profile experience so every public artist profile feels like a polished native iOS app, works cleanly from high-traffic fan entry points, passes mobile/in-app-browser QA, uses modern Next.js best practices, and has no visible jank, legacy UI mismatch, inconsistent flows, redundant copy, broken scrolling, or weird navigation behavior.

Public profiles are the **highest-traffic, highest-visibility, highest-acquisition surface** in the app. Treat this like production infrastructure plus premium consumer UX.

## Context

* Public profiles are shared by artists and visited by fans from social links, music releases, merch links, email, Gmail, SMS, search, and other inbound traffic.
* The current profile system was built, repurposed, redesigned, and patched over time. Expect mixed legacy components, inconsistent navigation patterns, inconsistent drawers/full pages/embeds, old design tokens, duplicate copy, and route-specific hacks.
* New visual design direction is mostly approved. **Do not start a new redesign.** Harden, normalize, polish, and make the implementation feel intentional.
* **Mobile-first.** Desktop must work, mobile is the core surface.
* **Gmail test:** every public-profile URL and deep-link URL must open cleanly from Gmail on iOS and Android.

## Primary Outcome

A fan should be able to open any public artist profile, tap through top-level sections, subscribe to alerts, follow links, view music/merch/social sections, and return/back out without encountering any jank, horizontal scroll, redundant UI, mismatched legacy design, broken drawer, clipped bottom content, weird copy, console error, or dead end.

## Non-Negotiables

1. **Mobile native feel** — bottom tab bar visually stable, thumb-friendly, safe-area-aware, consistent across top-level routes; one central surface config.
2. **Full-page consistency** — alerts/subscribe/email use one canonical full-page pattern; no drawer/embed variants for the same task.
3. **No scrolling jank** — no horizontal scroll at any mobile width, no content clipped behind tab bar/home indicator, no nested scroll traps, keyboard never obscures inputs.
4. **No legacy visual mismatch** — current design system/tokens used consistently, legacy components removed.
5. **No redundant or embarrassing copy** — short, clear, consumer-grade, consistent labels.
6. **Next.js best practices** — App Router conventions, layouts/templates, server components for data, `next/link`, Next metadata APIs, `next/image`, appropriate caching/revalidation.
7. **Accessibility and tap quality** — accessible names, active tab state, focus states, large touch targets, form labels/validation/states.
8. **High-traffic reliability** — no console errors, no hydration warnings, no broken images, no layout shifts, graceful fallback for missing data.

## Acceptance Criteria

**Product/UX:**

- [ ] All public profile top-level sections share one consistent shell.
- [ ] Bottom tab bar appears only where the spec says it appears, hides only where the spec says it hides.
- [ ] Alert/subscribe/email capture uses one canonical full-page pattern.
- [ ] No route uses a legacy drawer/embed for the same canonical task.
- [ ] No visible legacy design mismatch remains on public profile routes.
- [ ] No redundant or placeholder copy remains.
- [ ] Empty, loading, success, and error states look intentional.

**Mobile/layout:**

- [ ] Zero horizontal page scroll on tested devices.
- [ ] No clipped content behind bottom tab bar, browser chrome, keyboard, or home indicator.
- [ ] No broken long-name/long-link layouts.
- [ ] No scroll traps.
- [ ] Forms are usable with mobile keyboard open.

**Technical:**

- [ ] Typecheck, lint, unit/component tests, E2E/mobile tests all pass.
- [ ] No console errors or hydration warnings in tested flows.
- [ ] Dead legacy components removed or clearly isolated.
- [ ] Route config and surface spec committed.

**Performance:**

- [ ] LCP <= 2.5s at p75 on mobile.
- [ ] INP <= 200ms at p75.
- [ ] CLS <= 0.1 at p75.
- [ ] No avoidable layout shift from images, embeds, fonts, or tab bar transitions.

**Share/SEO:**

- [ ] Profile URLs have correct metadata.
- [ ] OG/Twitter images work.
- [ ] Canonical URLs are correct.
- [ ] Gmail/mobile link opening works.
- [ ] Unpublished/private/missing profiles behave intentionally.

* Do not allow separate agents to independently invent navigation or subscribe-flow behavior.

## Operating Principle

This is complete when the profiles feel **boringly solid**: beautiful, fast, consistent, touch-friendly, shareable, and impossible to embarrass the company under real fan traffic.

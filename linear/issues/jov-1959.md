## Follow-up

`apps/web/app/api/audience/visit-token/route.ts` exposes a public GET endpoint that mints valid tracking tokens for any `profileId` supplied by the caller. Because `getClientTrackingToken()` in `apps/web/lib/analytics/tracking-token.ts` signs the supplied profileId without further authentication, any caller who knows a profile UUID can generate a valid token and submit synthetic `/api/audience/visit` events without ever loading the profile page.

This was flagged by CodeRabbit (comment r3164378653 on PR #7986). The file is pre-existing and unchanged by PR #7986 — this is not a regression introduced by that PR, but it surfaces as a standing concern.

Suggested fix (from CodeRabbit): Move token issuance inside the visit handler (`/api/audience/visit`), or bind issuance to an authenticated page request (verify Origin/Referer + a server-side nonce or session cookie) so tokens cannot be minted with just a profileId.

## Why it matters

Allows artificially inflating profile view counts by submitting synthetic visit beacons. Could distort analytics and mislead creators about their actual reach.

## Classification

Candidate

## Acceptance criteria or triage question

Should token issuance be moved server-side into the visit handler, or is rate-limiting/Origin validation sufficient mitigation for the current traffic volume? Pickup agent must first judge whether to implement, close, or split this.

## Dependency

None — pre-existing file, not blocked by PR #7986.

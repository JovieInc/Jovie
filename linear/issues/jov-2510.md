# JOV-2510: Candidate follow-up: codex agent must post agent-run-artifact gate evidence on every PR

## Source

* Current issue: ad-hoc
* Source PR: 2026-05-21 `/drain` session
* Source branch/session: itstimwhite/zagreb drain workflow

## Follow-up

The codex agent opens draft PRs on `codex/drain-*` branches but never writes the `<!-- agent-run-artifact ... -->` JSON comment containing `verificationGates` for `gstack.qa.exhaustive`, `gstack.review`, and `gstack.ship`. This makes `apps/web/scripts/check-agent-gate-evidence.ts` fail every time in `.github/workflows/agent-pr-verify-ready.yml`.

Fix in the codex agent's PR-creation flow:

* After running `/qa --exhaustive`, `/review`, `/ship`, build an `AgentRunArtifact` whose `verificationGates` reflect each gate's `status` (`passed`/`failed`/`skipped`) with an `evidenceUrl` (CI run URL is fine) or `summary` and `checkedAt` ISO timestamp.
* Serialize via `formatAgentRunArtifactComment` from `apps/web/lib/agent-os/gate-evidence.ts` and post it as either the PR body or a PR comment BEFORE marking the PR ready.
* The `sourceRunId` in the artifact MUST equal the PR head SHA at the time of evaluation (this is what `AGENT_RUN_SOURCE_RUN_ID` is set to in the workflow).

## Why it matters

The Verify Draft Agent PR check exists to enforce that agent-opened PRs actually went through the gstack QA + review + ship gates. During this drain, 24 codex PRs slipped through because the check is not a required status (only PR Ready, Migration Guard, Fork PR Gate are). They were bulk-merged via the GitHub merge queue. If the codex agent posted gate evidence correctly, the workflow could be promoted to a required status and bad agent PRs would be auto-blocked.

## Classification

Candidate

## Triage question

Pickup agent must first judge whether to implement, close, or split this. Specifically: is the right fix to teach the codex agent to post artifact comments, OR to relax `check-agent-gate-evidence.ts` to accept lighter evidence (e.g., CI green is enough), OR to scrap the gate entirely?

## Dependency

None — this is a tooling cleanup, no other PR blocks it.

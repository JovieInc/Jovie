# JOV-2115: Admin operational control panel — component promotion prerequisite then Ops card

- **Status:** Todo
- **Priority:** 🟡 Medium
- **Assignee:** Unassigned
- **Team:** JOV
- **Project:** —
- **Labels:** area:ui, area:ops
- **URL:** https://linear.app/jovie/issue/JOV-2115/admin-operational-control-panel-component-promotion-prerequisite-then
- **Updated:** 2026-05-30T23:36:38.512Z

Daily-use operational toggles (waitlist on/off, campaign defaults, dev toolbar) currently live in `features/dashboard/organisms/` — importing into an admin feature would violate `eslint-plugin-boundaries` feature boundary rules.

**Surfaces:** `apps/web/components/features/dashboard/organisms/SettingsAdminSection.tsx`, `WaitlistSettingsPanel.tsx`, `CampaignSettingsPanel.tsx`

## Required two-PR sequence

1. **PR A:** Promote `WaitlistSettingsPanel` and `CampaignSettingsPanel` from `features/dashboard/organisms/` to `components/features/admin/` (or `components/organisms/`) — fixes boundary violation as prerequisite.
2. **PR B:** Create consolidated operational control panel card on the Ops screen using promoted components.

## Context dependency

Verify `WaitlistSettingsPanel` context dependencies before moving — it may depend on `useDashboardData` context not available on the Ops page.

## Specific toggles

* Dev Toolbar
* Waitlist on/off (via `WaitlistSettingsPanel`)
* Campaign defaults (via `CampaignSettingsPanel`)

Controls need loading, saving, and optimistic-failure states. Settings page post-removal: empty sections removed, not left blank.

**Prerequisite:** The control safety audit issue (filed separately) should be reviewed before this ships.

## Acceptance criteria

* All daily-use toggles accessible from one card on Ops.
* No `eslint-plugin-boundaries` violation.
* Settings no longer contains operational controls.
* PR A (promotion) merges before PR B (Ops card).
